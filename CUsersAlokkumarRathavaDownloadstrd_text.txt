1. Introduction and Scope
This document defines the complete technical requirements for SmartOPD — a full-stack clinical operating system for Indian hospitals. It covers backend architecture (NestJS), web frontend (Next.js with Atomic Design methodology), Android mobile application (Jetpack Compose), system architecture, all data dictionaries, API contracts, and integration specifications.
1.1 Purpose
This TRD serves as the single source of truth for engineering teams building SmartOPD. It defines what to build, how to structure it, and the data contracts each layer must honour.

2.3 Multi-Tenancy Architecture
SmartOPD is a multi-tenant SaaS. Each hospital (tenant) is isolated at the database layer using PostgreSQL Row-Level Security (RLS). Every table that contains tenant-scoped data includes a facility_id column. RLS policies enforce that queries only return rows belonging to the authenticated tenant.
sql
-- RLS policy example (PostgreSQL)
ALTER TABLE visits ENABLE ROW LEVEL SECURITY;
CREATE POLICY facility_isolation ON visits
USING (facility_id = current_setting('app.current_facility_id')::uuid);
-- Set in NestJS middleware for every request
await queryRunner.query(
`SET LOCAL app.current_facility_id = '${facilityId}'`
);
2.4 Offline Architecture
Hospital WAN connections are unreliable. SmartOPD handles offline scenarios at three layers:
Jetpack Compose apps use Room DB as an offline-first cache. All clinical data writes go to Room first, then sync to the backend via WorkManager when connectivity is restored.
Next.js web app uses a Service Worker to cache critical screens (nurse vitals form, basic prescription pad) for read-only operation during outages.
NestJS backend uses BullMQ with Redis-backed persistence. Failed ABDM API calls are retried with exponential backoff. FHIR bundle publishing is idempotent.

3.2.9 CRMModule
Intelligence layer that builds patient profiles from clinical data and drives retention workflows.
TypeScript — NestJS
// crm-segment.service.ts
async segmentPatient(patientId: string): Promise<PatientSegment[]> {
const visits = await this.visitRepo.findByPatient(patientId);
const diagnoses = visits.flatMap(v => v.consultation?.icd10Codes || []);
const segments: PatientSegment[] = [];
// Chronic disease tagging from ICD-10
if (diagnoses.some(code => DIABETES_CODES.includes(code)))   segments.push('CHRONIC_DIABETIC');
if (diagnoses.some(code => CARDIAC_CODES.includes(code)))    segments.push('CHRONIC_CARDIAC');
if (diagnoses.some(code => HYPERT_CODES.includes(code)))     segments.push('CHRONIC_HYPERTENSIVE');
// Recency segmentation
const daysSinceLastVisit = differenceInDays(new Date(), visits[0]?.startedAt);
if (daysSinceLastVisit > 90) segments.push('LAPSED');
if (daysSinceLastVisit < 30 && visits.length === 1) segments.push('NEW_PATIENT');
// Value segmentation
const ltv = await this.billRepo.sumByPatient(patientId);
if (ltv > 25000) segments.push('VIP_HIGH_VALUE');
return segments;
}

phone: patient.phone,
templateName: 'follow_up_reminder',
components: [{ type:'body', parameters:[{ type:'text', text: patient.name }] }]

intercept(ctx: ExecutionContext, next: CallHandler): Observable<any> {

5.3.2 VitalsViewModel
Kotlin — ViewModel
// VitalsViewModel.kt
@HiltViewModel
class VitalsViewModel @Inject constructor(
private val saveVitalsUseCase: SaveVitalsUseCase,
private val checkCriticalVitalsUseCase: CheckCriticalVitalsUseCase,
savedStateHandle: SavedStateHandle
) : ViewModel() {
private val visitId = savedStateHandle.get<String>("visitId")!!
private val _uiState = MutableStateFlow(VitalsUiState())
val uiState: StateFlow<VitalsUiState> = _uiState.asStateFlow()
fun updateSystolic(value: Int) {
_uiState.update { it.copy(vitals = it.vitals.copy(systolicBP = value)) }
checkCriticals()
}
private fun checkCriticals() {
val alert = checkCriticalVitalsUseCase(_uiState.value.vitals)
_uiState.update { it.copy(criticalVital = alert) }
}
fun submitVitals() = viewModelScope.launch {
_uiState.update { it.copy(isSubmitting = true) }
saveVitalsUseCase(_uiState.value.vitals.copy(visitId = visitId))
.onSuccess { _uiState.update { it.copy(isSubmitting = false, saved = true) } }
.onFailure { e -> _uiState.update { it.copy(isSubmitting = false, error = e.message) } }
}
fun notifyDoctor(alert: CriticalVitalAlert) = viewModelScope.launch {
vitalsRepository.notifyDoctor(visitId, alert)
}
}

8. Security Requirements
8.1 Authentication and Authorisation
JWT access tokens (15-minute TTL) + Refresh tokens (7-day TTL, rotating, Redis-blacklisted on logout)
All refresh tokens stored as a hashed value in PostgreSQL (refresh_tokens table) and keyed in Redis for O(1) blacklist lookups
Role-Based Access Control (RBAC) enforced at the Guard level on every endpoint via @Roles() decorator
Tenant isolation enforced at database layer via PostgreSQL Row-Level Security — not just application-layer filtering
ABDM gateway tokens cached in Redis with 23-hour TTL and auto-refreshed 5 minutes before expiry
8.2 Data Protection — DPDP Act 2023
Face embeddings stored as 512-dimensional vectors in pgvector — raw photos are never persisted
All embeddings encrypted at rest using AES-256 column-level encryption (pgcrypto)
Patient ABHA numbers encrypted at rest
Soft-delete with biometric_deletion_requested_at timestamp — embeddings zeroed within 24 hours of request via a scheduled job
Processor agreements (hospital ↔ SmartOPD) required before onboarding — tracked in the facilities table
Incident response playbook documented and tested quarterly

8.4 Network Security
All HTTP traffic enforced to HTTPS (TLS 1.2+). HTTP redirected to HTTPS at load balancer.
CORS configured to allowlist known frontend origins only. Wildcard CORS is never used.
Helmet middleware sets all recommended HTTP security headers (CSP, HSTS, X-Frame-Options).
Rate limiting: 100 req/min per IP globally, 10 req/min on /auth/login, 5 req/min on /auth/otp/send.
ABDM API calls made only from backend (never directly from client). Backend holds ABDM credentials.
VPC security groups allow backend ECS tasks to reach RDS and ElastiCache only — no public DB access.
